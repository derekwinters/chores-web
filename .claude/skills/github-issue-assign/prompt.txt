You are managing a GitHub issue implementation workflow for the chores-web repository.

When invoked with /github-issue-assign <issue-number>, follow this exact flow:

## Step 0: Verify Ready Status
- Fetch issue: gh issue view <number>
- Check if issue has `ready-for-work` label
- If label NOT present, STOP and inform user: "Issue not labeled ready-for-work. Cannot proceed with assignment."
- Otherwise, continue to Step 1

## Step 1: Branch Preparation
- Fetch issue details: gh issue view <number>
- Determine commit type from issue context:
  - "fix:" for bug fixes / reported issues
  - "feat:" for new features / enhancements
  - "refactor:" for code improvements / restructuring
  - "docs:" for documentation changes
  - "test:" for test-only changes
- Update main: git checkout main && git pull
- Create branch: git checkout -b <type>-issue-<number>
  (e.g., fix-issue-41, feat-issue-46, refactor-issue-50)
- Confirm branch created and ready

## Step 2: Implementation Guidance
- Show user the issue title and description
- Ask what specific changes they want to make
- Implement changes based on user requirements
- Update tests to cover new functionality
- Keep implementation focused on issue scope

## Step 3: Test Verification
- Run: npm test (in frontend directory)
- Verify all tests pass
- Report test results to user
- If tests fail, debug and fix before proceeding

## Step 4: Docker Rebuild
- Run: docker compose up --build -d
- Wait 5 seconds for containers to start
- Confirm all containers running

## Step 5: Manual Review & Approval
**This is the approval gate before committing.**

- Show simple summary of changes:
  - List files modified
  - Brief description of what changed in each file
  - Don't show full diffs (user reviews in browser)
- Inform user: "Containers rebuilt and running. Ready for manual testing."
- Ask user decision:
  - "Do you want to make more changes, or can I commit to the repository?"

If user wants MORE CHANGES:
- Go back to Step 2 (Implementation Guidance)
- Repeat cycle: implement → test → rebuild → review

If user APPROVES COMMIT:
- Proceed to Step 6

## Step 6: Commit
- Show user the full changes (git diff)
- ASK FOR FINAL CONFIRMATION before committing
- If approved: Stage changes: git add -A
- Create conventional commit with determined type
- Format: <type>: <description> (#<number>)
  where <type> is: feat, fix, refactor, docs, test, etc.
- Include Co-Authored-By footer

## Step 7: Push and PR
- ASK FOR CONFIRMATION before pushing
- If approved: Push branch: git push -u origin <type>-issue-<number>
- ASK FOR CONFIRMATION before creating PR
- If approved: Create PR: gh pr create --title "<type>: ..." --body "..."
  where <type> matches the commit type
- **IMPORTANT: Include "Closes #<number>" in PR body** to auto-close the issue
- Include summary, test plan, and link to issue
- Return PR URL to user

## Important Rules
- Use conventional commit format (feat:, fix:, refactor:, etc.)
- Always include issue number in commit/PR
- Run tests before creating PR
- Use docker compose up --build -d (not separate build/down/up)
- Ask user for approval at manual review step (Step 5)
- Support multiple iterations if user wants to make more changes
- Keep changes focused on issue scope
- Update tests as part of implementation

## Caveats
- Some issues may require multiple commits
- Complex issues may need breaking into subtasks
- Always verify Docker containers running before declaring done
- Test thoroughly in browser after Docker rebuild
- Loop back to implementation if user requests changes after review
