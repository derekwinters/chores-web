When invoked with /tdd <feature>, guide Test-Driven Development:

## Step 1: Red — Write Failing Test

1. Clarify behavior:
   - If feature description is vague, ask user for expected behavior
   - Get concrete examples of what should work

2. Identify test location:
   - Frontend: frontend/src/__tests__/<ComponentName>.test.jsx
   - Backend: backend/tests/test_<feature>.py
   - Create file if it doesn't exist

3. Write test:
   - Write the SMALLEST test that captures the behavior
   - Use existing test patterns in the codebase
   - Don't write multiple tests in one cycle

4. Verify failure:
   - Run: cd frontend && npm test (or cd backend && pytest)
   - Confirm test fails for RIGHT reason (not syntax/import error)
   - Report the failure to user

## Step 2: Green — Minimum Implementation

1. Write code:
   - Write ONLY enough to make the test pass
   - No extra features, edge cases, or abstractions
   - Follow existing code patterns

2. Verify passing:
   - Run test suite again
   - Confirm test now passes
   - Report success to user

## Step 3: Refactor

1. Review:
   - Look for duplication between test and code
   - Check for clarity and naming
   - Identify any extractable logic

2. Refactor if needed:
   - Make changes incrementally
   - Run tests after each change
   - Maintain green state throughout
   - Do NOT add new behavior

3. Confirm:
   - Tests still pass
   - Code is cleaner

## Iteration

1. Ask: "More behaviors to test for this feature?"
2. If yes, repeat from Step 1
3. If no, feature complete

## Important Rules

- ALWAYS write test first
- MINIMUM viable code per test
- GREEN state never broken
- No scope creep during refactor
- One behavior per cycle
